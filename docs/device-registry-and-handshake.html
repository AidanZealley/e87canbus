Device registry and CAN handshake

Proposed specification for research and phased implementation.

Status: Proposed design

Purpose

Introduce a common registration and heartbeat protocol for intelligent custom devices. The controller must know which supported devices are compatible and communicating without assuming every optional device is installed.

  • button_pad — NeoTrellis CAN button pad.
  • servotronic_controller — optional controller for additional Servotronic assistance; not the steering rack and not required for normal hydraulic PAS.

Decisions already made

Naming and LED state

Rename the ambiguous steering_controller to servotronic_controller consistently across code, contracts, UI, tests, docs, and protocol names. Keep buttons.led_colours as the one canonical controller-requested LED state. Remove normal-UI use of observed_led_colours.

Optional-device policy

  • Button pad and Servotronic controller are optional.
  • Their absence, staleness, incompatibility, or local fault does not make the controller globally unready.
  • Button-pad controls require an active compatible pad.
  • Additional Servotronic controls require an active compatible controller plus their relevant fresh inputs, such as vehicle speed.
  • Vehicle telemetry and ordinary /car operation do not require either custom device.
When the Servotronic device is absent or stale, the controller sends no additional-assistance command. Ordinary hydraulic PAS is outside this software capability model.

Passive vehicle telemetry

Vehicle ECUs are not registry participants. The controller passively observes configured signals and tracks valid, stale, or never_observed freshness. It does not probe arbitrary vehicle CAN IDs by default.

Architecture

Capability catalogue and runtime registry

Code/configuration statically defines supported roles, accepted versions and capabilities, instance policy, heartbeat policy, command codec, and feature dependencies. The controller owner thread maintains a non-persisted in-memory registry of currently accepted devices.

Supported roles
  button_pad
  servotronic_controller

Registry entry
  role
  stable device ID
  device boot/session ID
  protocol version and capability bits
  lifecycle status
  last valid heartbeat
  last device-reported local fault/status

Use a provisioned 16-bit numeric ID in v1. The runtime identity is (role, stable device ID, current boot/session ID). Configuration allow-lists expected role/ID pairs. Unknown or duplicate identities never silently grant a feature.

Lifecycle

Controller registry
no registration → not_found
valid HELLO → pending
valid acknowledged heartbeat → active
heartbeat timeout → stale / unavailable
unsupported version/capability → incompatible
device-reported/detected issue → fault

Device
booting → discovering → operational
                 ↘ incompatible / local fault
operational → controller_lost → discovering

For the pad: use a breathing pattern while discovering, normal controller-requested LEDs while operational, and a local red error pattern after bounded controller loss or local fault.

Protocol behaviour

Device broadcasts HELLO while discovering
  → Controller validates role, ID, version, capabilities
  → Controller replies WELCOME / heartbeat acknowledgement
  → Device becomes operational and sends heartbeat/status
  → Controller retains an active registry lease while fresh

The controller listens for this explicit custom-device protocol; it does not scan arbitrary CAN traffic. Acknowledgements/beacons let a device know a controller is listening. Version, identity, session, and sequence data must reject stale traffic after either side restarts.

MessageRequired fields
HELLOProtocol version, role, stable device ID, device boot/session ID, capability bits, sequence.
WELCOMEProtocol version, role, device ID, device boot/session ID, controller boot/session ID, sequence/echo.
HEARTBEATIdentity/session fields, sequence, optional compact local fault/status code.

Final layouts fit standard 8-byte CAN payloads and document IDs, byte order, malformed handling, and compatibility rules. Starting research defaults: roughly 1 Hz cadence with jitter, stale after three missed intervals, bounded retry/backoff, and browser state publication only on lifecycle transitions.

State, API, and UI projection

Publish a read-only registry-oriented device projection: configured identity, source mode, lifecycle status, protocol/capability summary, and last transition/fault summary. Preserve references for unchanged entries so one device’s update does not redraw unrelated UI.

  • /car shows capability-specific not-found, unavailable, or incompatible status.
  • /dev exposes registry state and simulation-only fault scenarios.
  • Normal LED rendering always uses buttons.led_colours.
  • Backend command paths reject feature commands when required active devices or fresh inputs are absent.

Simulation requirements

Virtual devices implement the same protocol and state machines as physical peers. The controller consumes protocol frames and registry transitions, not a simulator-only success path. The simulator must deterministically model absent devices, delayed boot, successful registration, incompatible version/capabilities, heartbeat loss, reboot, compact local faults, and malformed frames. Tests use a fake clock; the virtual vehicle remains a passive telemetry producer.

Verification expectations

  • Registration, re-registration, reboot, timeout, malformed, incompatible, unknown, and duplicate-identity paths.
  • Optional-device feature gating in backend and UI.
  • Simulation and live-contract projection behaviour.
  • No lifecycle update/render storm under heartbeat traffic.
  • Controller-canonical LEDs and separation of optional-device availability from process health/readiness.

Handoff questions for planning

  1. Final public status names and registry live-event schema.
  2. Exact CAN arbitration IDs and 8-byte layouts.
  3. Firmware role/device-ID provisioning workflow.
  4. Final heartbeat, timeout, retry, duplicate-instance, and fault policies.
  5. Exact /car status placement and /dev fault controls.
  6. Safe rename sequence from steering_controller to servotronic_controller.
